# 레포지토리 계층
* 데이터베이스에 접근하여 CRUD 작업을 수행하는 계층
    * **C**reate: 데이터 생성
    * **R**ead: 데이터 조회
    * **U**pdate: 기존 데이터 수정
    * **D**elete: 데이터 삭제
* 서비스 계층은 레포지토리를 통해 데이터를 생성하고 조회하고 수정하고 삭제하는 모든 동작을 수행
* DB 전달 과정
    1. 애플리케이션 서버는 DB가 이해할 수 있는 SQL을 연결된 커넥션을 통해 DB에 전달
    2. DB는 전달된 SQL을 수행하고 그 결과를 응답
    3. 애플리케이션 서버는 결과 활용
## ORM과 JPA
* JDBC: Java에서 DB에 접근하기 위한 표준화된 API. SQL을 직접 다룸
    * 단점: SQL 코드 관리하기 까다롭다. 객체지향 패러다임도 불일치
* ORM (Object-Relationial Mapping)
    * 객체와 관계형 데이터베이스의 데이터를 자동으로 매핑(연결)해주는 것
    * 객체를 메서드로 조작하면 SQL을 실행해주기 때문에 개발 생산성 향상
* JPA (Java Persistence API)
    * 자바 진영의 ORM 기술 표준
    * 객체를 DB에 저장하고 관리하기 위한 인터페이스와 기능을 제공하는 API
    * 대표적 구현체: Hibernate
    * 자바 앱과 JDBC API 사이에서 동작함. 내부적으로 JDBC 사용
## JPA의 동작 방식
* EntityManager
    * 엔티티 객체를 DB와 연결해주는 창구 역할
    * 등록 -> `em.persist()`
    * 조회 -> `em.find()`
    * 삭제 -> `em.remove()`
    * EM은 엔테티를 영속성 컨텍스트에 저장
    * 엔티티: 데이터베이스에 저장할 객체, 테이블과 1:1 매핑되는 객체
* 영속성 컨텍스트
    * JPA에서 엔티티를 관리하기 위해서 만들어져있는 가상 공간, EM 통해 접근
    * 생명주기: 비영속/영속/준영속/삭제
    * 특징: 1차캐시, 동일성 보장, 트랜잭션을 지원하는 쓰기 지연, 변경 감지
    * 트랜잭션: DB에서 수행되는 작업의 단위. 여러 연산 중 모두 성공시 커밋하여 반영, 아니면 롤백. 예) 주문정보저장->재고차감->포인트적립
    * 트랜잭션을 지원하는 쓰기 지연: 트랜잭션 완료(커밋)시점에 한번에 묶어서 DB에 보내는 구조. 네트워킹 횟수 줄여 성능 최적화 가능
    * 등록 시: 영속성 컨텍스트에 저장했다가 커밋 시 DB에 저장
    * 조회 시: 한번 조회하여 영속성 컨텍스트에 캐시된 객체는 DB를 거치지 않고 바로 재사용. 트랜잭션 내에선 같은 엔티티 조회 시 같은 객체 그대로 반환 (동일성 보장)
    * 수정 시: 커밋할 때 엔티티의 최초 상태와 현재 상태를 비교해서 변경됐으면 자동으로 UPDATE 쿼리 보내 DB에 반영 (변경 감지-Dirty Checking. 영속 상태의 엔티티에만 적용됨)
    * 삭제 시: 영속성 컨텍스트에서 객체를 삭제 대상으로 표시하고 더 이상 관리X, 삭제 SQL 저장해놨다가 flush/커밋 시 전달
* JPQL
    * SQL과 비슷하지만, 테이블이 아니라 엔티티 객체를 기준으로 쿼리 작성
    * 복잡한 조회일 경우 사용
